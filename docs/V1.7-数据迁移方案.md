# V1.6 → V1.7 数据迁移 / 演示环境清理方案

> 未经用户明确确认，不执行任何云数据库清空、覆盖或 initDemo。

## 原则

1. 部署前必须备份。
2. 保留 `admins`、`users`、`config`。
3. 正式环境禁止 `initDemo` 覆盖真实业务数据。
4. 演示环境可清理八个业务集合后由超管可信任务跑 `initDemo`。

## 备份清单（部署前）

导出或快照以下集合：

- 保留型：`admins`、`users`、`config`
- 业务型：`products`、`skus`、`categories`、`stations`、`batches`、`batchStations`、`batchInventory`、`orders`、`refunds`、`refundRequests`、`verificationLogs`、`deliveryWindows`、`carts`、`openGroupNudges`、`businessDays`、`placementLogs`、`contactLogs`、`operationLogs`、`notificationOutbox`、`paymentEvents`、`runtimeLocks`

记录：备份时间、操作人、对应 Git SHA。

## 路径 A：正式/有真实订单

1. 维护窗口，停止商家发布新批次。
2. 备份全库。
3. **不删业务数据**；部署同版云函数与小程序。
4. 人工核对字段兼容：
   - 订单需有 `phoneTail`、`pickupQrToken`（老单若无，核销走人工）
   - 成团按 `paidUserCount`；老数据若仅有按件计数，需评估是否重算或仅对新批次生效
   - `subscribeGroupResult` / `subscribePickupNotice` 缺省按未授权
5. 配置 `config/system` 模板 ID、确认 `SUPER_ADMIN_OPENID`。
6. **不跑 initDemo**。

## 路径 B：纯演示环境（无真实资金）

1. 备份（仍建议做）。
2. 清空业务八集合（及 V1.7 扩展集合），**不动** admins/users/config。
3. 同版部署云函数。
4. 受信任务执行 `{"action":"initDemo"}`（仅超管/受信）。
5. 上传同版小程序，MOCK 真机冒烟。

## 路径 C：回滚

1. 恢复备份集合。
2. 部署上一稳定 Git SHA 的云函数 + 小程序。
3. 验证：下单预占、按人计数、核销权限。
