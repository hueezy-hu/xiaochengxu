### 1.1 本版相对 V1.6 的主要变更

以下为 V1.7 相对 V1.6 的规则改动与新增，均为本轮需求对齐后的最终决定：

1. 待支付预占时长由 15 分钟改为 **3 分钟**；预占为后台短暂状态，用户端不再有可返回续付的“待支付订单”列表。
2. 成团门槛由“每站已付款 5 件（按件）”改为“每站 **5 个不同付款用户（按人）**”；库存仍按件核算，成团判定按人。
3. 成团判定时刻明确为 **22:00 截单即锁定**：截单时该站满 5 人即锁定为成团成功，成团后个别退款只减人数、不散团，退到 0 人才自动取消该站；不设“维持线”自动退团，极端塌方由超级管理员手动取消。
4. 核销凭证由“6 位自取码”改为 **手机尾号后 4 位**；核销支持“扫二维码”和“输尾号”双路径，手动输尾号为兜底保险；同一场次出现相同尾号时后台提醒管理员。
5. 新增 **交付拍照上传**为必做动作：到自取点/核销点、以及无人取货挪货时均需拍照上传，用户可见。
6. 明确两种交付模式并 **按站点配置**：有人核销、无人放置。
7. 退款改为 **两段制**：交付完成前用户自助即时退款（不审核）；交付完成后仅能“申请退款”，转商家人工个例处理。
8. 商家批次发布流程细化：站点/窗口/地点/图片/核销模式为平时维护好的固定资料，发布时只选 SKU、填库存、勾选启用站点；发布截止 10:00 为软提醒；新增“今日休息”态与“催开团计数”。
9. 权限保持两级（超级管理员 / 普通管理员），普通管理员能力固定为核销加拍照加查看，可被超级管理员分配到多个站点。
10. 订阅消息由 1 个模板增加为 **2 个模板**（成团结果通知、自提提醒），支付成功时一次性授权两个模板。

## 2. 产品目标

泰斓 TAILAN 是一个面向家庭烘焙泰式斑斓甜品的微信小程序，采用“今天拼、明天取”的地铁站集中自提模式。

V1.7 要解决四个核心问题：

1. 用户能快速了解品牌、选择商品和自提站点，并完成拼团下单。
2. 商家能按每日生产能力配置各 SKU 库存、勾选启用站点，并手动发布“今天卖、明天取”批次。
3. 系统能正确处理 3 分钟预占、每站 5 人成团、22:00 截单锁定、次日 12:00 配送确认以及两段制退款。
4. 商家和核销员能按站点完成备料、拍照、核销、迟到联系和固定地点放置交付。

### 2.1 本版本做什么

- 品牌展示首页，含“今日休息 / 催开团”状态。
- 商品分类、商品详情和 SKU 选择。
- 购物车暂存选购，结算时选一个站点生成一单，并保留“立即购买”快捷路径。
- 商家手动发布“今天卖、明天取”批次，一天一个在售批次。
- 每个 SKU 独立设置批次库存，所有站点共享、先到先得。
- 每站累计 5 个不同付款用户达到成团门槛。
- 待支付订单预占库存 3 分钟。
- 当天 22:00 停止新下单并对该站锁定成团结果。
- 取货日 12:00 按站点最终确认是否配送。
- 用户在订单完成交付前可以整单自助退款；交付后仅能申请人工退款。
- 站点、自提时间、地点说明、图片、核销模式管理。
- 手机尾号核销、二维码扫码、输尾号手动核销、交付拍照和固定地点放置。
- 超级管理员、普通管理员（核销员）权限隔离，普通管理员支持多站。
- MOCK 支付与退款流程。

### 2.2 本版本不做什么

- 不接入真实微信支付。
- 不支持部分退款，只支持整单退款。
- 不支持订单顺延或迁移到下一批次。
- 不做质量问题申请、图片举证和质量售后审核系统。
- 不做团长奖励、周期购和复杂营销优惠。
- 不自动发布每日批次，必须由商家确认库存后手动发布。
- 不做预约制（今天只销售明天取货批次）。
- 催开团只做计数展示，不做实时推送。

## 3. 角色与权限概览

### 3.1 普通用户

- 可以不登录浏览首页、分类和商品详情。
- 付款前必须填写昵称和有效手机号；手机号存档复用、可修改。
- 可以选择商品、数量和站点。
- 可以发起支付并完成 MOCK 支付（预占为后台状态，用户无“待支付订单列表”）。
- 可以查看已付款订单、取货券、二维码和手机尾号取货凭证。
- 可以在订单完成交付前自助整单即时退款。
- 交付完成后若确实未取到，可以提交退款申请，转商家人工个例处理。
- 可以把取货券截图、二维码交给可信任的人代取。
- 首页非开团态可点击“催开团”，仅累计计数。

### 3.2 超级管理员 `superAdmin`

- 管理商品、SKU、价格和上下架状态。
- 管理站点池、线路、出口、地点说明、图片和核销模式。
- 维护站点自提日期、开始时间、结束时间等固定资料。
- 创建、编辑和手动发布批次；发布时选 SKU、填库存、勾选启用站点。
- 发布后可追加库存；可下架 SKU（默认已付订单照常履约，可选对已付订单发起退款）。
- 设置“今日休息”，或在未发布时查看催开团计数。
- 低于 5 人时手动确认配送；成团后塌方时手动取消配送。
- 手动关闭整个批次或单个站点。
- 处理退款重试与交付后的人工个例退款。
- 查看所有站点备料和订单。
- 执行普通核销和异常跨站核销。
- 管理普通管理员权限，并为其分配一个或多个授权站点。

### 3.3 普通管理员（核销员）`verifier`

- 只能查看被超级管理员授权的一个或多个站点。
- 可以查看对应站点备料清单和待核销订单。
- 可以扫二维码或输入手机尾号核销对应站点订单。
- 可以联系对应站点迟到用户。
- 必须在到点/放置时拍照上传交付照片。
- 可以上传固定地点放置证明。
- 不能管理商品、库存、批次、站点、退款，也看不到未授权站点。
- 能力集合固定为“核销 + 拍照 + 查看授权站点订单”，不做逐项开关；本版本不设第三种自定义角色。
- 不能普通跨站核销。

## 4. 核心业务参数

| 参数 | V1.7 规则 |

| 销售模式 | 今天卖，明天取；不做预约制 |

| 发布截止 | 建议销售日 10:00 前发布，软提醒，过点仍可发 |

| 成团判定时刻 | 22:00 截单即锁定 |
| 成团后退款 | 只减人数不散团，退到 0 人才自动取消该站 |
| 维持线 | 不设自动维持线；塌方由超级管理员手动取消 |
| SKU 库存 | 各 SKU 独立设置，批次内所有站点共享、先到先得 |

| 待支付是否计入成团 | 不计入 |
| 用户待支付列表 | 无；预占为后台短暂状态，用户无法返回续付 |
| 最终配送确认 | 取货日北京时间 12:00 |
| 12:00 不足 5 人 | 取消该站并退还全部剩余订单 |
| 12:00 达到 5 人 | 确认配送 |
| 低于 5 人手动配送 | 允许，仅超级管理员，二次确认并填写原因 |
| 成团后跌破 5 人 | 只要还有有效订单，仍继续配送 |
| 成团后退款至 0 人 | 自动取消该站配送 |
| 交付模式 | 按站点配置：有人核销 / 无人放置 |
| 核销凭证 | 手机尾号后 4 位 |
| 核销方式 | 扫二维码 或 输入手机尾号；手动为兜底 |
| 尾号重复 | 同场次相同尾号后台提醒管理员，核销时列出多单人工区分 |

| 完成交付 | 已核销；或无人放置模式商家点“已放置”并上传照片 |
| 未取货 | 有人核销：保持待取，挪货拍照，可延后自取，商家手动收尾，不退款；无人放置：放置即完成 |
| 迟到 | 窗口内迟到仍算正常取货，照常核销 |
| 代取 | 支持截图、二维码或手机尾号代取 |
| 手机号 | 付款前强制填写；个人主体手动，非个人主体一键授权 |

所有日期、22:00、12:00 和 3 分钟超时计算必须使用北京时间 UTC+8。

## 6. 首页：品牌概念展示页

### 6.1 页面定位

首页只负责建立品牌认知和引导用户进入点单，不承担站点选择、拼团进度管理或复杂操作。

### 6.2 首页保留内容

1. 泰斓 TAILAN 品牌主视觉。
2. 简短品牌理念和泰式斑斓甜品概念。
3. 主打产品或原料氛围图片。
4. 简短说明“今天拼·明天取”。
5. 唯一主按钮“立即去拼”。
6. 底部四项 tabBar。

### 6.3 首页删除内容

- 删除完整批次流程卡。
- 删除布吉站、大学城站等站点卡片。
- 删除各站已拼人数和进度条。
- 删除各站时间、出口和地点说明。
- 删除首页直接选站能力。
- 删除首页长规则条和复杂交易说明。

“我的订单”通过底部 tabBar 进入，“联系商家”放在我的页和取货券页。

### 6.4 首页营业状态

首页存在三种状态：

- **开团中**：当天已发布在售批次，“立即去拼”正常进入点单并可下单。
- **今日休息**：商家主动点击“今日休息”。首页显示正经的休息提示（如“今日休息，明天见”），点单页与商品一律显示“售罄”，不可下单。
- **未开团（商家在忙）**：商家既未发布也未点休息。首页显示轻松的催单话术（如“老板摸鱼中，踢一脚催他开团”）与“催开团”按钮，点单页与商品一律显示“售罄”。

非开团态下页面不禁用、可浏览，只是无法下单。首页“催开团”按钮仅累计计数，见 9.6。

## 7. 点单与商品页面

### 7.1 点单/商品分类页

页面只保留：

- 当前批次简要状态条（开团中 / 售罄）。
- 明确取货日期。
- 当天 22:00 截止提示。
- 商品分类竖栏。
- 商品图片、名称、描述、起售价和剩余库存。
- 商品详情入口。

页面不展示完整站点卡、站点进度条、出口和地点说明。非开团态时商品一律显示“售罄”。

### 7.2 商品详情页

- 展示商品图片轮播。
- 展示中文名、泰文名、分类、描述和标签。
- 展示 SKU、规格、单价和批次剩余库存。
- 用户选择数量。
- 数量不能超过当前可售库存。
- 底部展示合计和“选择自提站点”。

### 7.3 商品与 SKU

演示菜单沿用 5 款真实商品：

1. 蝶糯桑卡雅，¥6/个。
2. 香斓花糕，¥15/盒。
3. 椰丝洛冲，¥15/盒。
4. 叶盏西米糕，¥26/盒。
5. 斑斓椰肉团子，¥18/盒。

分类：

- 糯香经典
- 斑斓软糕
- 清爽凉品

已付款订单使用 `orders.items` 中的名称、规格和价格快照。商家后续改价不能影响历史订单。

### 7.4 购物车

购物车是选购暂存区，不占库存、不影响成团，真正的预占、选站、成团、退款都在结算之后，完全复用既有规则。

- **加入与暂存**：用户在商品详情选好 SKU 和数量后加入购物车；购物车按用户服务端持久化，换设备/重进仍在；同商品不同 SKU 各占一行。
- **入口**：tabBar 四项不变，购物车为非 tab 页，在点单页/商品详情以带角标的悬浮图标进入。
- **购物车页**：商品列表、数量加减、逐项勾选、全选/反选、删除或滑动删除、合计只统计勾选项、去结算显示勾选件数。
- **不占库存**：加入购物车只查是否可卖，不预占；预占只发生在发起支付时。
- **价格**：购物车存加入时价用于展示降价提示；最终价格快照在支付时锁定，写入 `orders.items`。
- **失效处理**：售罄、下架或不属于当前接单批次的条目置为失效，置灰沉底、不可结算，提供“清除失效”。22:00 后、次日新批次或休息态时，旧批次条目一律失效，用户在新批次重新添加。
- **一单一站**：购物车不绑定站点；结算时为勾选的这批商品选择一个自提站点、生成一单。去两个站取货需分两次结算。
- **结算与校验**：结算走“商品勾选 → 选择站点 → 填写/确认手机号 → 发起支付”；发起支付时对勾选的多个 SKU 在同一事务一次性预占 3 分钟，任一 SKU 库存不足则整单不预占并提示是哪件、让用户改数量。
- **成团与退款**：一次结算即一单，在该站仍只计 1 人成团；多 SKU 一单整单退款，不做部分退。
- **快捷路径并存**：商品详情保留“立即购买/立即去拼”，跳过购物车，单 SKU 直达选站与支付。

## 8. 站点池与自提信息管理

站点、自提窗口、地点说明、图片和核销模式均为平时维护好的固定资料，超级管理员随时可编辑；发布批次时不再重复填写，只勾选启用哪些站点。

### 8.1 站点池基础字段

超级管理员可以新增、编辑、启用或停用站点。每个站点至少包含：

- `name`：站点名称。
- `line`：地铁线路。
- `exit`：出口。
- `pickupNote`：基础位置说明。
- `defaultLocationImages[]`：默认位置图片，最多 3 张。
- `verifyMode`：交付模式，取值“有人核销”或“无人放置”。
- `arriveAt` / `leaveAt`：默认自提开始与结束时间。
- `status`：启用或停用。

主打站点为：

- 布吉站，3/14 号线。
- 大学城站，5 号线。

### 8.2 批次启用站点

商家发布批次时，从站点池勾选本批次开放的站点。每站的取货日期默认取发布日次日；窗口、地点说明、图片、核销模式沿用站点池资料，如需针对本批次微调可覆盖：

- `pickupDate`：取货日期（默认 D+1）。
- `arriveAt` / `leaveAt`：自提开始与结束时间。
- `locationNote`：本批次位置说明。
- `locationImages[]`：本批次位置图片，最多 3 张。
- `verifyMode`：本批次核销模式（默认取站点设置）。

本批次覆盖值优先于站点池默认信息。成团门槛统一取默认 5 人，非逐站填写。

### 8.3 图片上传

- 管理员通过微信原生选择图片并上传云存储。
- 后台保存云文件 `fileID`。
- 图片上传失败不能发布不完整批次。
- 编辑时保留已上传图片，支持删除和补传。
- 用户在选择站点页和取货券页查看图片。
- 图片不写入二维码；二维码只用于识别订单。

### 8.4 用户选择站点页

每张站点卡展示：

- 站点名、线路和出口。
- 已付款人数和 5 人门槛进度（“已拼 X/5 人”）。
- 当前状态。
- 自提日期、开始时间和结束时间。
- 详细位置说明。
- 位置图片缩略图，点击可预览。

用户必须选择一个开放站点才能支付。

## 9. 批次发布、休息态与销售日规则

### 9.1 发布内容

商家在销售日做一次“发布明日批次”的操作，只需：

1. 勾选本批次上架的 SKU。
2. 为每个 SKU 填写本批次库存件数（即产量上限）。
3. 勾选本批次启用哪些站点（从已维护好的站点池中选）。

站点窗口、地点、图片、核销模式、成团门槛均为既有资料或默认值，不在发布时重复填写。

### 9.2 手动发布

- 批次只能由超级管理员手动发布。
- 发布前显示完整预览和二次确认。
- 发布日为 D 时，取货日期必须是 D+1。
- 截单时间自动设为 D 日北京时间 22:00。
- 同一时间最多存在一个用户可见的接单中批次；一天只销售一个批次。
- 商家不发布时，用户只能浏览商品，不能下单。
- 不自动复制昨天库存，也不自动发布下一批。

### 9.3 发布截止与休息态

- 建议在销售日 10:00 前发布，10:00 为软提醒；过了 10:00 商家仍可发布，只是用户当天可购时长变短。
- 商家可主动点击“今日休息”，首页进入“今日休息”态，当天不开团。
- 商家既未发布也未点休息时，首页进入“未开团（商家在忙）”态，展示催单话术与“催开团”按钮。
- 两种非开团态下，点单页与商品一律显示“售罄”，不可下单。

### 9.4 发布后修改

- **追加库存**：允许随时上调某 SKU 库存，无副作用。
- **下架 / 减少 SKU**：允许，采用 b+c 结合：
  - 默认（b）：已付款订单照常履约，仅停止该 SKU 新购。
  - 可选（c）：该 SKU 确实无法制作时，商家可对其已付款订单主动发起退款。
- 修改必须记录操作人和时间；涉及已付款订单的下架需二次确认。

### 9.5 22:00 后规则

- 当天 22:00 立即停止新订单创建和支付。
- 已创建但尚未付款的预占全部超时，释放库存。
- 22:00 对每个站点按 5 人门槛锁定成团结果，见第 11 节。
- 22:00 后不销售后天取批次。
- 第二天必须由商家重新确认库存并手动发布新的“今天卖、明天取”批次。

### 9.6 催开团计数

- 首页“未开团（商家在忙）”态展示“催开团”按钮。
- 用户每次点击累计一次，去重按 openid 计当日一次。
- 首页向用户展示累计人数（如“今天已有 8 人在等你开团”）。
- 商家后台可见该计数，不做实时推送；本版本不接入商家端订阅或客服消息。

### 9.7 发布前服务端校验

必须满足：

- 至少勾选一个上架 SKU，且每个 SKU 库存为非负整数或明确标记无限库存。
- 同一 SKU 不重复。
- 至少勾选一个启用站点，同一站点不重复。
- 每个启用站点的窗口、地点说明、核销模式完整（来自站点池或本批次覆盖值）。
- 结束时间晚于开始时间。
- 每站位置图片不超过 3 张。
- 取货日期是发布日次日。

批次及其站点、窗口和库存必须全有或全无。推荐先完整写入草稿，全部成功后再切换为接单中。

## 10. 库存、待支付预占与支付确认

### 10.1 共享库存模型

库存单位为 `batchId + skuId`。例如：

- A 商品本批库存 20 件。
- B 商品本批库存 8 件。

布吉站和大学城站购买时共同扣减这 20 件和 8 件，先到先得，不拆分站点库存。因此可能出现某站抢得多、另一站剩得少的情况，属预期行为。

### 10.2 库存字段

- `totalQty`：本批设定生产总量。
- `availableQty`：当前还可被新订单预占的数量。
- `reservedQty`：预占中订单临时占用数量。
- `soldQty`：已付款数量。
- `refundedQty`：已付款后退款数量，用于经营统计。
- `isUnlimited`：是否为无限库存。

必须满足：

`availableQty + reservedQty + soldQty - refundedQty = totalQty`

无限库存不参与有限库存售罄判断。

### 10.3 发起支付与 3 分钟预占

用户点击“发起支付”前，系统必须在事务中：

1. 校验批次仍在接单中。
2. 校验当前时间早于 22:00。
3. 校验站点仍开放。
4. 校验手机号完整。
5. 校验 SKU 和数量。
6. 预占库存。
7. 创建预占中订单。
8. 写入 `expiresAt = createdAt + 3 分钟`。

购物车结算时，勾选的多个 SKU 在同一事务一次性预占；任一 SKU 库存不足则整单不预占并提示是哪件。

预占成功后：

- `availableQty` 减少，`reservedQty` 增加。
- 不增加站点已付款人数与件数。
- 不触发成团。

预占为后台短暂状态，用户端不展示“待支付订单列表”、不能返回续付。正式支付模式下，微信支付下单必须设置订单有效期 `time_expire = 3 分钟`，与预占超时对齐，保证超时后微信侧也拒绝该笔支付。

### 10.4 支付成功

3 分钟内支付成功后，在事务中：

- `reservedQty` 减少，`soldQty` 增加。
- 订单进入待配送确认。
- 站点 `paidItemCount` 增加商品件数（用于备料）。
- 站点 `paidUserCount` 按付款用户 openid 去重后累加（用于成团判定）。
- 判断是否达到 5 人门槛。

同一支付回调重复到达时，只能处理一次（幂等）。

迟到成功兜底：若预占已因超时被释放，但支付回调随后成功（钱已到账），系统必须重新尝试为该订单抢占库存——抢得到则补记为已付款，抢不到则立即对该笔发起原路退款。该兜底保证超时时长设置为多少都不会超卖。

### 10.5 取消、超时和支付结果不明确

- 用户主动取消支付，立即释放预占库存。
- 3 分钟未付款，订单进入已超时并释放库存，前端提示“支付超时，请重新发起”。
- 用户重新发起支付时生成全新订单号，重新预占、重新计时。
- 定时任务持续清理超时订单；查询库存、创建订单和支付确认时也要惰性清理已过期预占。
- 支付结果不明确时先查订单状态，不重复创建订单。
- 22:00 到达时，所有预占中订单立即超时并释放库存。

### 10.6 售罄

- 单个 SKU 售罄只关闭该 SKU。
- 其他 SKU 仍有库存时，批次继续接单至 22:00。
- 只有所有有限且可售 SKU 都为 0，才触发批次库存售罄。
- 库存售罄后停止新订单，并进入与手动截单相同的批次推进流程。

## 11. 每站 5 人成团与配送确认

### 11.1 成团计数

- 成团按站点统计，单位为“不同付款用户数（openid 去重）”。
- 一个用户在同一站点无论下几单、买几件，成团只计 1 人。
- 库存与备料仍按件核算，与成团人数是两套并存的口径。
- 预占中订单不计入成团。
- 默认门槛为 5 人。

示例：布吉站有 5 个不同用户各自付款，即达门槛；1 个用户买 5 件不算成团。

### 11.2 销售期间状态

- 已付款人数少于 5：`拼团中`，站点卡展示“已拼 X/5 人”。
- 已付款人数达到 5：`已成团待确认`。
- 达到门槛后仍可继续下单，直到 22:00 或库存售罄。
- 销售期间用户退款后，重新计算已付款人数。
- 截单前人数退款归零，站点仍保持拼团中，后续用户可以重新发起该站拼团。

### 11.3 22:00 截单锁定

22:00 停止新增订单，并逐站按 5 人门槛锁定结果：

- 满 5 人：站点锁定为 `已成团待确认`。用户当晚即可看到“拼团成功，明日到 XX 站凭手机尾号自取”。
- 不足 5 人：站点进入 `未成团待处理`。默认将在取货日 12:00 关闭并退款；超级管理员可在 12:00 前手动确认配送挽救。用户当晚看到“暂未成团，若最终未开团将于明日 12:00 前全额退款”。
- 所有预占中订单在 22:00 超时释放。
- 用户仍可对未完成交付的订单退款；不再允许新增订单补回人数。

### 11.4 取货日 12:00 配送确认

系统在取货日北京时间 12:00 逐站处理。已由超级管理员手动确认配送或手动取消的站点，12:00 任务不再按人数重新判定，直接保留其状态。

#### 已成团（达到 5 人或已手动确认）

- 站点进入 `已确认配送`。
- 相关订单进入 `待自提`。
- 商家按站点备料并前往配送。
- 用户端展示明确自提时间、地点和图片。

#### 未成团（不足 5 人且未手动确认）

- 站点进入 `关闭退款中`。
- 剩余有效订单全部整单退款。
- 退款完成后站点进入 `已关闭`。
- 商家不需要前往该站。

### 11.5 低于 5 人手动确认配送

超级管理员可以在 12:00 前手动确认低于 5 人的站点仍然配送。操作要求：

- 二次确认，展示站点和当前已付款人数。
- 必须填写原因，记录操作人和时间。
- 操作后站点立即进入已确认配送，订单进入待自提。
- 取货日 12:00 自动任务不得再次把该站判定为不足 5 人关闭。

### 11.6 成团后手动取消（对称能力）

对已锁定成团的站点，如因大量退款塌方到不划算配送，超级管理员可在 12:00 前手动取消该站配送。操作要求：

- 二次确认，展示站点当前有效人数与件数。
- 必须填写原因，记录操作人和时间。
- 对该站全部剩余有效订单原路退款，并推送通知。
- 本操作是 11.5 的反方向能力，用于替代自动“维持线”，把塌方判断交给人工。

### 11.7 12:00 后用户退款

- 用户仍可以在订单完成交付前整单退款。
- 退款后减少站点已付款人数并记录退款件数。
- 如果还有至少 1 个有效订单，站点保持已确认配送，商家继续交付，不因跌破 5 人而取消其他用户。
- 如果所有用户都退款、有效人数变为 0，系统自动取消该站配送。

## 12. 用户资料与手机号

### 12.1 浏览与下单边界

- 用户无需提供资料即可浏览。
- 付款前必须填写昵称和有效手机号。
- 手机号为空或格式不正确时，不能创建订单、预占库存或参加拼团。

### 12.2 手机号方案（双逻辑）

手机号在下单前强制取得，兼作取货核销凭证（尾号后 4 位），因此必须真实、可联系。

- 个人主体（`MANUAL_PHONE = true`）：默认用户手动输入手机号，不依赖微信一键获取。
- 非个人主体（认证小程序）：切换为微信 `getPhoneNumber` 一键授权，代码侧预留开关，与真实支付同一门槛一并升级。
- 一键授权失败时仍需保留手动输入兜底。

### 12.3 手机号存档复用

- 用户填写过一次后存档，下次下单自动带出。
- 用户可在下单前修改手机号。

### 12.4 手机号用途与可见范围

手机号仅用于：自提联系、迟到联系、订单和地点异常通知、取货核销匹配。

权限：

- 超级管理员可以查看订单完整手机号。
- 被授权站点核销员可以查看或拨打本场用户手机号。
- 其他页面和其他站点默认脱敏。

### 12.5 我的页（个人中心）

沿用电商小程序个人中心骨架，按本项目裁剪：

- **顶部**：头像 + 昵称（`chooseAvatar` / `nickname`）；未设置时显示“点击设置头像昵称”。
- **订单入口**：进入我的订单（与 tabBar 冗余入口，常规做法）。
- **功能列表**：联系商家、手机号管理（查看与修改存档手机号，见 12.3）、使用规则、关于泰斓。
- **身份入口**：当前 openid 若为超级管理员或核销员，则在“我的”页按身份显示“商家工作台 / 现场核销”入口；普通用户不可见。这是商家端唯一进入方式，不走隐藏链接。

## 13. 用户退款机制（两段制）

### 13.1 基本原则

- 交付完成前：用户可自助整单即时退款，不需商家审核。
- 交付完成后：用户不能自助退款，只能提交退款申请，转商家人工个例处理。
- 不支持部分商品退款；不支持顺延或迁移到下一批。
- 交付前全程可退，由此产生的备料损失由商家承担（本项目明确选择用户友好优先）。

### 13.2 交付前自助退款

可自助退款的状态：

- 待配送确认。
- 待自提。
- 已确认配送但尚未核销、尚未完成固定地点放置。

用户点击“整单退款”后立即进入退款流程；MOCK 模式直接置为已退款；预占中订单不走退款，直接取消并释放预占。

### 13.3 交付后申请退款

- 已核销或已完成固定地点放置的订单，视为已交付，不再支持用户自助退款。
- 用户如确实未取到，可提交“退款申请”。
- 申请进入商家/超级管理员待处理列表，由人工个例判断，可人工退款或拒绝并说明。
- 无人放置模式下未取到的用户，也走此申请通道。

### 13.4 退款处理流程

退款必须使用统一流程：

1. 校验订单属于当前用户（自助）或属于人工审批范围（申请）。
2. 校验退款前置条件（交付前自助 / 交付后需人工）。
3. 使用订单 ID 生成稳定退款幂等键。
4. 把订单标记为退款处理中。
5. 回退已付款人数与件数，恢复批次库存账目。
6. MOCK 模式直接进入已退款。
7. 正式支付模式调用微信支付退款接口。
8. 正式支付以退款通知或退款查询结果为最终结果。
9. 失败时记录退款失败并支持使用相同退款单号重试。

### 13.5 退款与核销并发

退款和核销必须互斥：

- 退款事务先成功：取货凭证立即失效，不能核销。
- 核销事务先成功：订单已完成，不能自助退款（只能走交付后申请）。
- 固定地点放置事务先成功：订单完成交付，不能自助退款。

### 13.6 商家无法交付

商家无法交付时只支持退款，不支持顺延。场景包括：商家临时取消站点、无法按承诺到达、商品缺失。超级管理员关闭站点或取消配送后，所有未完成订单进入退款流程。

### 13.7 当前版本不做质量售后

V1.7 不提供独立的质量问题申请、图片举证和商家质量审核流程。用户仍可通过“联系商家”沟通，但不在本版本建立结构化质量售后状态机。

## 14. 支付成功、订阅提醒与分享

### 14.1 支付成功页

展示：支付结果、商品和金额、站点名称、当前站点已付款人数、“已成团待确认”或“拼团中”状态、分享按钮、开启自提提醒、查看订单、继续购买。

### 14.2 订阅提醒（2 个模板）

订阅消息使用两个一次性订阅模板，在支付成功页由用户一次授权（一个授权弹窗同时请求两个模板）：

- **模板一 · 成团结果通知**：取货日 12:00 配送确认后发送。成团成功发“成团成功，请 X 点到 XX 站凭手机尾号自取”；未成团发“未成团，已全额退款”。时间与地点必须与该站窗口设置一致。
- **模板二 · 自提提醒**：自提窗口开始前发送“您在 XX 站的订单可取啦”类取货提醒。

授权后回写 `orders.subscribeGroupResult` 与 `orders.subscribePickupNotice`。未授权或发送失败不能阻塞订单主流程。

### 14.3 分享（基础分享，Level 1）

- **入口**：支付成功页、取货券页、站点团页均可分享“该站拼团”。
- **分享带参**：分享 path 携带 `batchStationId`，可选携带 `inviterOpenid` 作为邀请埋点（仅记录、不返佣，为 V2 团长激励预留字段）。
- **落地页 `groupPage`**：展示该站“已拼 X/5 人”进度、取货日期时间地点、简要规则；“我也要拼”进入点单页，不在分享页锁定 SKU；已成团则显示“已成团”。
- 分享不是完成支付的强制条件。
- **归入 V2**：助力返佣、团长红包、阶梯价、虚拟成团数不在本版本实现（虚拟成团数默认关闭，如需可加配置开关）。

## 15. 取货券、代取与现场核销

### 15.1 取货券内容

- 站点名称、线路和出口。
- 自提日期和时间窗口。
- 详细位置说明和位置图片。
- 商品和数量、订单金额。
- 大号手机尾号后 4 位取货凭证。
- 二维码（用于扫码核销，只标识订单）。
- 联系商家按钮。
- 交付现场照片（商家上传后展示）。

### 15.2 截图代取

- 用户可以截图取货券，或把二维码、手机尾号告知可信任的人。
- 核销以有效订单为准，不强制核验代取人身份。
- 第一次成功核销后立即失效。
- 用户应自行保管，因主动泄露或重复转发产生的提前核销风险由用户承担。

### 15.3 普通核销（尾号 + 扫码双路径）

核销员先选择当前批次和站点，再用以下任一路径核销：

- **扫二维码**（主路径）：扫用户取货券二维码，系统定位订单。
- **输手机尾号**（兜底/手动）：输入用户手机尾号后 4 位，系统列出该场次匹配订单；命中多单时人工区分后选定。

后端必须检查：

- 订单存在，属于当前批次、当前站点团。
- 订单状态为待自提或已放置待自取。
- 订单没有退款，取货凭证没有核销过。
- 核销员具有该站点权限。

成功后：订单进入已完成，写入核销时间、核销人、核销方式（扫码/尾号/手动）、批次和站点日志。

### 15.4 尾号重复预警

- 成团锁定后，系统扫描每个场次已付款订单的手机尾号后 4 位。
- 同一场次出现相同尾号时，标记该场次并提醒管理员。
- 核销时若输尾号命中多单，界面列出全部候选订单，核销员按姓名、完整手机号或商品人工选定，避免误核。

### 15.5 交付拍照（必做）

- 核销员/商家到自取点或核销点，必须拍照上传交付现场照片（货已到、摆放位置），保存云存储，用户可在取货券和订单看到。
- 无人放置模式点击“已放置”时必须附现场照片。
- 未取货挪货到存放点时必须再拍一张，供用户凭图延后自取。

### 15.6 异常跨站核销

- 普通核销员不能跨站核销。
- 超级管理员可以异常跨站核销，必须展示正确站点与当前站点差异、二次确认并填写原因。
- 核销日志必须标记为异常跨站核销。

## 16. 迟到、未取货与双交付模式

### 16.1 交付模式按站点配置

每个站点在站点池设置 `verifyMode` 为“有人核销”或“无人放置”，批次可覆盖。两种模式的交付完成判定不同，直接决定“不可退”的起点。

### 16.2 有人核销模式

- 窗口内商家/核销员在现场，用户出示二维码或报手机尾号，核销员核销即完成交付。
- 窗口内迟到仍算正常取货，照常核销。

### 16.3 无人放置模式

- 商家把货放到预先约定的固定地点，点击“整场已放置”并上传现场照片。
- 该操作使全场订单一次性进入 `已放置待自取`、视为交付完成、转为不可自助退款。
- 无现场核验动作，用户凭照片自取。
- 用户未取到时通过 13.3 的申请通道走人工个例退款。

### 16.4 迟到联系

自提窗口内用户未到时，商家或本场核销员可通过订单手机号联系用户。后台展示手机号、拨打按钮、联系状态和备注。

### 16.5 有人核销的未取货（no-show）

窗口结束仍有未核销订单时：

- 订单保持 `待自提`，不自动退款（商家已履约到站）。
- 商家把货挪到存放点并拍照上传，用户凭照片延后自取。
- 商家当天清点后手动把这些订单收尾为 `已完成未取`，避免长期挂起（历史上出现过“未取货死路”）。
- 现做甜品易损，延后自取现实中通常仅当天有效。

### 16.6 交付责任

商家按约定完成联系、固定地点放置、拍照和系统记录后，视为完成交付。

- 交付后不支持用户自助退款，只能提交申请走人工个例。
- 不支持顺延。
- 商家不承诺持续看管，也不设置系统保留倒计时。
- 因用户迟取、现场管理或他人拿取产生的风险由用户承担。
- 固定地点必须经过允许并适合临时放置食品，不能随意占用地铁站公共区域。

### 16.7 结束本场

- 核销员只能结束被授权站点本场自提。
- 结束前必须处理所有待自提订单：已核销、已放置待自取或已完成未取。
- 不能直接把未处理订单批量变成未取货终态。
- 超级管理员可以异常提前结束，但必须二次确认并填写原因。

## 17. 商家工作台

### 17.1 工作台概览

展示：当前接单批次、已截单待配送确认批次、各站已付款人数和 5 人门槛、各站件数备料、各站 12:00 确认结果、待处理退款（含交付后个例申请）、当天自提和核销进度、待放置订单、今日营业状态与休息开关、催开团计数。

### 17.2 批次操作

- 发布批次（选 SKU、填库存、勾选站点）。
- 追加库存。
- 下架 SKU（b+c 结合）。
- 设置“今日休息”。
- 手动关闭整个批次。
- 查看操作记录。

### 17.3 站点操作

- 低于 5 人手动确认配送。
- 成团后手动取消配送。
- 修改自提窗口和位置图片。
- 关闭单个站点并退款。
- 标记到达、上传交付照片、进入备料和核销。

### 17.4 手动关闭

“关闭整个批次”和“关闭单个站点”是两个独立操作。共同要求：仅超级管理员、二次确认、必须填写原因、记录操作人时间批次站点、影响订单进入统一退款流程、重复操作必须幂等。

### 17.5 修改时间和地点

- 批次发布前可以自由修改。
- 已有付款订单后，必须填写修改原因。
- 修改日期、站点、出口或时间变化超过 30 分钟，视为重大变化，必须通知已付款用户。
- 如果商家无法按修改后的承诺履约，只能退款，不顺延。

### 17.6 商品与 SKU 管理

商品与 SKU 是“菜单库”，与批次发布分离：这里维护有哪些款，发布时从上架款里勾当日款并定量。仅超级管理员可操作。

- **商品(SPU)**：中文名、泰文名、分类、描述、标签、主图与轮播图、上下架状态、排序。
- **SKU**：规格名、单价、库存基准、状态、排序；一个商品可挂多个 SKU，现状每款一个 SKU。
- **分类**：糯香经典、斑斓软糕、清爽凉品可增删改，商品挂分类。
- **改价即时生效**：云开发自建、无审核流程；改价只影响之后订单，已付订单用 `orders.items` 快照、不受影响。
- **删除保护**：有历史订单引用的商品或 SKU 只能下架，不能硬删；无引用可删。
- **图片**：主图与轮播上传云存储；下架商品不进入批次发布的可选款列表，但不影响历史订单。

## 18. 状态机

### 18.1 批次状态 `batches.status`

| 状态 | 说明 | 退出事件 |
|---|---|---|
| 草稿 | 商家配置库存和站点，用户不可见 | 手动发布或删除 |
| 接单中 | 用户可创建订单，截止到 22:00 | 22:00、库存售罄、手动关闭 |
| 已截单待配送确认 | 停止新订单，各站已按人数锁定，等待 12:00 | 12:00 处理全部站点 |
| 配送进行中 | 至少一个站点已确认配送 | 全部站点完成或关闭 |
| 关闭退款中 | 商家关闭批次，正在退款 | 所有退款处理完成 |
| 已结束 | 全部站点完成或退款关闭 | 终态 |

当天无发布批次时，首页处于“今日休息”或“未开团”态，属营业状态而非批次状态。

### 18.2 站点团状态 `batchStations.status`

| 状态 | 说明 |
|---|---|
| 拼团中 | 已付款人数不足 5 |
| 已成团待确认 | 已达到 5 人（22:00 锁定或手动确认前的达标态） |
| 未成团待处理 | 22:00 不足 5 人，默认 12:00 关闭退款，可被手动挽救 |
| 已确认配送 | 12:00 达到 5 人或商家手动确认，商家承担交付承诺 |
| 关闭退款中 | 未成团或商家关闭/取消，正在退款 |
| 已关闭 | 有效订单为 0 或退款完成，不配送 |
| 自提进行中 | 商家已到达并开始核销/放置 |
| 已完成 | 本场所有有效订单已核销、已放置或已收尾 |

### 18.3 订单状态 `orders.status`

| 状态 | 说明 |
|---|---|
| 预占中 | 后台预占库存 3 分钟，用户无待支付列表，不计入成团 |
| 已取消 | 用户取消，库存已释放 |
| 已超时 | 3 分钟或 22:00 超时，库存已释放 |
| 待配送确认 | 已付款，等待站点 12:00 确认 |
| 待自提 | 站点已确认配送 |
| 退款申请待处理 | 交付后用户提交申请，等待商家人工处理 |
| 退款处理中 | 已发起整单退款 |
| 已退款 | 退款流程完成 |
| 已放置待自取 | 商家完成固定地点放置和拍照，视为已交付 |
| 已完成 | 已现场核销 |
| 已完成未取 | 有人核销窗口结束用户未取，商家挪货拍照后收尾，不退款 |

旧版的 15 分钟待支付、6 位自取码、截单过渡、顺延等状态不再作为 V1.7 订单状态；实现时必须统一迁移到本节定义的状态集合。

### 18.4 退款状态 `refunds.status`

| 状态 | 说明 |
|---|---|
| 退款申请待处理 | 交付后个例申请，等待人工审批 |
| 退款处理中 | MOCK 处理或等待微信退款结果 |
| 退款失败 | 调用失败，允许同退款单号重试 |
| 已退款 | MOCK 完成或微信确认成功 |

本版本没有用户质量售后审核，因此不使用待审核、已驳回等质量售后状态。

### 19.13 `users`

### 19.15 `carts`

- 用户 openid、批次 ID、条目列表（SKU、数量、加入时价、加入时间）、逐项勾选状态、失效标记。

## 20. MOCK 模式与正式支付上线门槛

### 20.1 当前 MOCK 规则

- `MOCK_PAY = true`，`MANUAL_PHONE = true`。
- 支付按钮模拟支付成功，退款模拟立即成功。
- 用于验证库存、成团、页面、核销和退款状态。
- 不代表已具备正式收款能力。

### 20.2 正式上线前必须完成

1. 取得可用微信支付商户号。
2. 前端真实拉起微信支付，下单设置 `time_expire = 3 分钟`。
3. 后端校验合法支付回调、商户号、订单号、金额和支付状态。
4. 支付回调幂等，并实现迟到成功兜底（重抢库存或自动退款）。
5. 接入微信支付退款 API，接收并校验退款结果通知。
6. 退款查询和失败重试，使用稳定退款单号避免重复退款。
7. 完成真实小额支付、取消、超时、退款和重复回调联调。
8. 非个人主体后，将手机号切换为 `getPhoneNumber` 一键授权（`MANUAL_PHONE = false`）。
9. 申请并配置两个订阅消息模板 ID（成团结果、自提提醒），完成正式发送器与失败重试联调。

以上事项必须长期保留在项目说明和上线前待办中，不能因为 MOCK 流程跑通而删除。

## 21. 异常和恢复规则

### 21.1 重复创建或支付

- 支付按钮请求期间禁用。
- 同一订单重复支付只处理一次。
- 支付结果未知时先查订单，不重新占库存。

### 21.2 预占释放失败

- 超时订单保持待清理标记。
- 定时任务反复重试。
- 释放必须幂等，不能增加超过总库存。

### 21.3 迟到支付回调

- 预占已释放但回调成功时，重新抢库存；抢到补记已付款，抢不到自动原路退款。
- 处理必须幂等，不重复退款或重复计入成团。

### 21.4 12:00 确认任务失败

- 批次保持已截单待配送确认。
- 记录未处理站点和错误。
- 定时器重试，已处理站点不重复处理。

### 21.5 批量退款中断

- 每个订单独立退款幂等。
- 站点保持关闭退款中。
- 所有订单完成后才进入已关闭。

### 21.6 通知失败

- 不影响订单主流程。
- 记录失败并允许管理员查看。
- 用户仍可在订单和取货券中查看最新信息。

### 21.7 图片上传失败

- 不保存无效 `fileID`。
- 发布批次前必须检查图片列表。
- 交付拍照上传失败允许重试，不丢失其他表单内容。

## 22. 验收标准

### 22.1 文档与文件完整性

- JS 逐文件语法检查无错误。
- JSON 全部可解析。
- UTF-8、BOM、NULL、文件截断和 WXML 标签扫描无损坏。
- 前端调用 action 与云函数路由一致。
- 页面不得出现旧版 15 分钟预占、6 位码、按件成团、单段退款或顺延入口。

### 22.2 首页与点单

- 首页不展示站点卡和站点进度。
- 首页只有一个主操作“立即去拼”。
- 首页正确切换开团/今日休息/未开团三态；非开团态商品显示售罄。
- 用户先选商品，再进入独立选站页。

### 22.3 批次发布

- 商家发布只需选 SKU、填库存、勾选启用站点。
- 只有超级管理员可以发布。
- 发布日与取货日相差 1 天；截止时间固定为发布日 22:00。
- 10:00 为软提醒，过点仍可发布。
- 同时只有一个用户可见接单批次；一天只销售一个批次。
- 追加库存允许；下架 SKU 支持 b+c；催开团只计数不推送。

### 22.4 站点与图片

- 站点池维护站点、线路、出口、窗口、说明、图片、核销模式。
- 选站页和取货券展示正确图片。
- 修改图片不会破坏已有窗口信息。
- 未完整配置的站点不能被批次启用。

### 22.5 待支付库存

- 发起支付立即预占库存，预占为后台状态、无用户待支付列表。
- 预占不计入成团。
- 3 分钟内支付成功，预占转已售。
- 3 分钟超时释放库存并提示重新发起。
- 22:00 时所有预占订单超时。
- 重复点击不重复占库存。
- 两个站竞争最后一份库存只能有一个成功。

### 22.6 成团与 12:00 确认

- 每站不同付款用户数达到 5 人时显示已成团。
- 一个用户多单多件仍只计 1 人。
- 22:00 逐站锁定成团结果。
- 取货日 12:00 达到 5 人确认配送，不足 5 人剩余订单全部退款。
- 超级管理员可低于 5 人手动确认配送，也可成团后手动取消，均需原因。
- 已手动确认配送的站点不会在 12:00 被重新取消。
- 12:00 后退款跌破 5 人但仍大于 0，继续配送；全部退至 0 人自动取消。

### 22.7 退款

- 交付前用户可自助整单即时退款，不需审核。
- 交付后仅能申请，转商家人工个例处理。
- 退款后减少站点人数与件数并记录退款库存。
- 退款与核销并发时只有一个操作成功。
- 同一订单重复退款只产生一笔有效退款。
- MOCK 模式直接进入已退款；正式支付以退款通知或查询结果为准。

### 22.8 手机号

- 用户不填手机号不能创建订单和预占库存。
- 手机号格式错误有明确提示；手机号存档复用可改。
- 个人主体手动输入；非个人主体切换一键授权。
- 核销员只能看到授权站点用户手机号。

### 22.9 核销与放置

- 核销支持扫码与输尾号，手动为兜底。
- 同场次尾号重复后台预警，命中多单时人工区分。
- 普通核销员只能核销授权站点订单；错站明确提示。
- 超级管理员跨站核销需要确认和原因；二次核销被拒绝。
- 到点/放置必须拍照上传，用户可见。
- 无人放置点“已放置”后订单转不可自助退。
- 有人核销窗口结束的未取货订单可收尾为已完成未取、不退款。
- 结束本场前不存在未处理待自提订单。

### 22.10 权限

- 普通用户不能绑定超级管理员。
- 普通管理员不能管理商品、库存、批次、站点或退款。
- 超级管理员可为普通管理员分配一个或多个授权站点。
- 整批关闭和单站关闭分开，均要求超级管理员、二次确认和原因。

## 23. 演示数据要求

`initDemo` 使用动态北京时间生成：

- 取货日期：初始化次日。

## 24. V1.7 决策记录

- 真正发起支付时才预占，预占 3 分钟；预占为后台状态，用户无待支付列表；真实支付以 `time_expire` 对齐并靠 payCallback 幂等 + 迟到兜底防超卖。
- 成团按每站不同付款用户数计，满 5 人成团；库存与备料按件另算。
- 22:00 截单即锁定成团结果；成团后个别退款不散团，退到 0 人才自动取消；不设自动维持线，塌方由超级管理员手动取消。
- 取货日 12:00 执行配送确认与关闭退款；超级管理员可低于门槛手动确认、也可成团后手动取消。
- 退款两段制：交付前自助即时退、交付后申请人工个例；交付前全程可退，损失商家承担；不部分退、不顺延。
- 核销凭证改为手机尾号后 4 位；扫码与输尾号双路径，手动为兜底；同场次尾号重复后台预警。
- 交付拍照为必做动作；两种交付模式按站点配置（有人核销 / 无人放置）。
- 有人核销未取货保持待取、挪货拍照、可延后自取、商家手动收尾、不退款；无人放置放置即完成。
- 批次发布：站点/窗口/图片/核销模式为固定资料，发布只选 SKU、填库存、勾选站点；一天一批、今天卖明天取、不做预约。
- 发布截止 10:00 为软提醒；今日休息与未开团两态；催开团只计数不推送；追加库存允许、下架 SKU 采用 b+c。
- 权限两级；普通管理员能力固定为核销加拍照加查看，可被分配到多个站点。
- 手机号双逻辑：个人主体手动、非个人主体一键授权，存档复用。
- 订阅消息 2 个模板：成团结果通知与自提提醒，支付成功时一次授权。
- 购物车为选购暂存区、服务端持久化、不占库存不碰成团；结算按“勾选商品 → 选一个站 → 一单”生成，多 SKU 整单退；保留立即购买快捷路径。
- 商品/SKU 为菜单库、与批次发布分离；改价无审核即时生效、只影响之后订单；有订单引用只能下架不能删。
- 分享做基础 Level 1（带参落地页 + 我也要拼 + 邀请埋点）；团长激励、助力返佣、虚拟成团数归 V2。
- 我的页含头像昵称、订单入口、联系商家、手机号管理、使用规则，并按身份显示商家/核销入口。
- 当前为内部 MOCK 演示版，真实支付、退款、一键手机号与订阅正式发送留作上线前必做项。
